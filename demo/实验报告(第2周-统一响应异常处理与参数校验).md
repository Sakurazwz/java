## 三、实验原理

在传统 Web 开发中，不同接口可能返回字符串、对象、集合或错误信息，导致前端处理复杂、格式不统一。  
为了提升系统可维护性和接口一致性，本实验采用统一响应结构，将所有接口返回值统一封装为 `Result<T>`。

统一响应的核心思想是：
- 成功和失败都使用同一 JSON 结构返回；
- 使用枚举统一管理响应码与响应消息；
- 在响应对象中预留时间戳、traceId 等扩展字段，便于日志追踪与接口调试。

当业务运行过程中出现异常时，若不做统一处理，异常会以默认格式返回，不利于前端展示，也可能暴露系统内部信息。  
因此，本实验通过 `@RestControllerAdvice` 配合 `@ExceptionHandler` 实现全局异常处理，将业务异常、参数校验异常、非法参数异常以及系统异常统一转换为规范响应。

参数校验方面，Spring Boot 集成了 Bean Validation 规范，可以通过注解方式对请求参数进行约束，例如：
- `@NotBlank`：非空白字符串校验
- `@Email`：邮箱格式校验
- `@Pattern`：正则表达式校验
- `@Min` / `@Max`：数值范围校验

同时，还可以通过自定义注解和校验器扩展校验能力，例如密码强度校验、手机号校验、枚举值校验等。

## 四、实验内容与步骤

### 1. 创建统一响应结构

#### 1.1 创建响应码枚举 `ResultCode`
定义系统统一的响应码，包括：
- 成功：200
- 客户端错误：400、401、403、404 等
- 服务端错误：500、503 等
- 业务错误：1001、1002、1003 等

#### 1.2 创建统一响应类 `Result<T>`
设计字段如下：
- `code`：响应码
- `message`：响应信息
- `data`：响应数据
- `timestamp`：时间戳
- `traceId`：链路追踪标识

并提供以下静态工厂方法：
- `success()`
- `success(T data)`
- `success(String message, T data)`
- `error(ResultCode resultCode)`
- `error(int code, String message)`
- `error(ResultCode resultCode, T data)`
- `error(String message)`

#### 1.3 创建用户实体类 `User`
定义用户的基本属性，包括：
- id
- username
- email
- phone
- password
- age
- status
- createTime
- updateTime

#### 1.4 编写测试控制器
创建 `UserController`，编写如下测试接口：
- `/user/get`
- `/user/list`
- `/user/empty`
- `/user/fail`
- `/user/custom-error`

### 2. 实现全局异常处理

#### 2.1 创建业务异常类 `BusinessException`
业务异常继承 `RuntimeException`，并携带：
- 错误码
- 错误消息

支持多种构造方式，便于在业务代码中直接抛出。

#### 2.2 创建全局异常处理器 `GlobalExceptionHandler`
使用 `@RestControllerAdvice` 编写统一异常处理器，分别处理以下异常：
- `BusinessException`
- `MethodArgumentNotValidException`
- `BindException`
- `IllegalArgumentException`
- `ArithmeticException`
- `NullPointerException`
- `Exception`

其中：
- 参数校验异常返回 400；
- 系统异常返回 500；
- 业务异常返回自定义错误码。

#### 2.3 编写异常测试接口
在 `UserController` 中增加：
- `/user/business-exception`
- `/user/business-exception-custom`
- `/user/arithmetic-exception`
- `/user/null-exception`
- `/user/illegal-argument`

用于验证异常处理是否生效。

### 3. 实现参数校验

#### 3.1 添加参数校验依赖
在 `pom.xml` 中加入 `spring-boot-starter-validation` 依赖。

#### 3.2 创建注册 DTO `UserRegisterDTO`
字段及校验规则如下：
- `username`：不能为空，长度 3-20，只能包含字母、数字和下划线
- `password`：不能为空，长度 6-20
- `confirmPassword`：不能为空
- `email`：不能为空，格式必须正确
- `phone`：手机号格式校验
- `age`：不能为空，范围 1-120
- `birthDate`：必须为过去日期
- `gender`：范围 0-2

#### 3.3 创建注册接口
在 `UserController` 中编写：
- `POST /user/register`

接口逻辑：
1. 使用 `@Valid @RequestBody` 触发参数校验；
2. 判断两次密码是否一致；
3. 返回统一格式的注册结果。

#### 3.4 测试参数校验场景
使用 cURL/Postman 测试以下情况：
- 正常注册
- 用户名为空
- 用户名长度不足
- 邮箱格式错误
- 手机号格式错误
- 年龄超出范围
- 多个参数同时错误

### 4. 实现 `@RequestParam` 与 `@ModelAttribute` 校验

#### 4.1 RequestParam 校验
在 `UserController` 类上添加 `@Validated`，并编写：
- `GET /user/query?username=xxx`

对 `username` 进行非空和长度校验。

#### 4.2 ModelAttribute 校验
创建 `UserQueryDTO`，并编写：
- `GET /user/search`

对 `username`、`pageNum`、`pageSize` 进行参数约束。

### 5. 创建自定义校验注解

#### 5.1 密码强度校验 `@PasswordStrength`
要求密码必须：
- 包含大写字母
- 包含小写字母
- 包含数字
- 长度为 8-20 位

实现：
- 自定义注解 `PasswordStrength`
- 校验器 `PasswordStrengthValidator`

并将其应用到 `UserRegisterDTO.password` 字段中。

#### 5.2 手机号校验 `@PhoneNumber`
创建自定义手机号校验注解及校验器，用于替代或增强正则校验。

#### 5.3 枚举值校验 `@EnumValue`
创建性别枚举 `Gender`，并实现枚举值校验注解：
- 输入值必须属于枚举定义范围
- 支持字符串枚举名校验

### 6. 实现分组校验

#### 6.1 定义分组接口
创建 `ValidationGroup`，定义：
- `Create`
- `Update`
- `Delete`

#### 6.2 创建更新 DTO `UserUpdateDTO`
根据不同分组设置不同校验规则：
- 新增用户时，用户名、邮箱必填；
- 更新用户时，ID 必填；
- 年龄字段在多个分组中都校验。

#### 6.3 编写分组校验接口
在 `UserController` 中添加：
- `POST /user/create`
- `POST /user/update`

分别使用不同的校验分组。

### 7. 配置快速失败模式

#### 7.1 创建校验配置类 `ValidationConfig`
配置 `hibernate.validator.fail_fast=true`，实现第一个校验失败后立即返回。

#### 7.2 测试快速失败
对比开启前后多个错误场景的返回结果：
- 开启前：返回所有错误信息；
- 开启后：只返回第一个错误信息。

### 9.分组解密`Create`和`Update`

- 已定义`ValidationGroup.Create`和`ValidationGroup.Update`分组。
- 在创建用户时，使用`Create`分组校验，用户名、邮箱等必填字段不能为空。
- 在更新用户时，使用`Update`分组校验，用户ID为必填项。

**测试现象：**

- `Create`分组下，如果不传用户名和邮箱，会返回：
  - `用户名不能为空`
  - `邮箱不能为空`
- `Update`分组下：
  - 只传`id`时，校验通过
  - 不传`id`时，会返回：
    - `用户ID不能为空`

**结论测试：**
分组校验能够根据不同的业务场景正确触发不同的规则，创建和更新接口的校验逻辑符合预期。

###  10.业务异常增强

- `BusinessException`已用于业务异常统一发布。
- 在接口调用中出现以下情况时，会引发业务异常：
  - 用户名已存在
  - 邮箱已被注册
  - 用户名或密码错误
  - 账号已被禁用
- 从代码结构来看，异常处理已支持记录更详细的业务信息，除了排查问题。

**测试结论：**
业务异常能够正确拦截并返回统一错误信息，提升了接口解决性和排错效率。

## 第一部分：AOP 基础概念（20分钟）

### 什么是 AOP？

**AOP（Aspect-Oriented Programming）** 面向切面编程，是一种编程范式，通过**预编译**方式和**运行期动态代理**实现程序功能的统一维护。

**简单理解**：将多个模块中的公共功能（如日志、事务、权限）提取出来，统一管理。

---

### 传统方式 vs AOP

**场景**：在每个接口执行前后记录日志。

#### 传统方式（冗余）

```java
@PostMapping("/user/register")
public Result<String> register(@RequestBody UserRegisterDTO dto) {
    log.info("开始注册用户: {}", dto.getUsername());
    long startTime = System.currentTimeMillis();

    try {
        // 业务逻辑
        userService.register(dto);

        long endTime = System.currentTimeMillis();
        log.info("注册用户成功，耗时: {}ms", endTime - startTime);
        return Result.success();
    } catch (Exception e) {
        long endTime = System.currentTimeMillis();
        log.error("注册用户失败，耗时: {}ms", endTime - startTime, e);
        throw e;
    }
}

@PostMapping("/user/login")
public Result<String> login(@RequestBody UserLoginDTO dto) {
    log.info("开始登录用户: {}", dto.getUsername());
    long startTime = System.currentTimeMillis();

    try {
        // 业务逻辑
        userService.login(dto);

        long endTime = System.currentTimeMillis();
        log.info("登录用户成功，耗时: {}ms", endTime - startTime);
        return Result.success();
    } catch (Exception e) {
        long endTime = System.currentTimeMillis();
        log.error("登录用户失败，耗时: {}ms", endTime - startTime, e);
        throw e;
    }
}
```

**问题**：
- 代码重复
- 业务逻辑和日志代码混合
- 维护困难

---

#### AOP 方式（优雅）

```java
// 业务代码（简洁）
@PostMapping("/user/register")
public Result<String> register(@RequestBody UserRegisterDTO dto) {
    userService.register(dto);
    return Result.success();
}

// AOP 切面（统一处理日志）
@Aspect
@Component
public class LogAspect {

    @Around("@annotation(org.springframework.web.bind.annotation.PostMapping)")
    public Object logAround(ProceedingJoinPoint joinPoint) throws Throwable {
        // 记录请求日志
        log.info("请求: {}", joinPoint.getSignature());
        long startTime = System.currentTimeMillis();

        try {
            Object result = joinPoint.proceed();  // 执行目标方法
            long endTime = System.currentTimeMillis();
            log.info("执行成功，耗时: {}ms", endTime - startTime);
            return result;
        } catch (Exception e) {
            long endTime = System.currentTimeMillis();
            log.error("执行失败，耗时: {}ms", endTime - startTime, e);
            throw e;
        }
    }
}
```

**优势**：
- 业务代码简洁
- 日志逻辑集中管理
- 易于维护和扩展

### AOP 核心概念

| 术语 | 说明 | 示例 |
|------|------|------|
| **JoinPoint（连接点）** | 程序执行的某个位置（如方法调用、异常抛出） | 所有的 Controller 方法 |
| **Pointcut（切入点）** | 匹配连接点的表达式（哪些方法需要被拦截） | `execution(* com.example.controller.*.*(..))` |
| **Advice（通知）** | 在切入点执行的代码（何时执行） | 前置通知、后置通知、环绕通知 |
| **Aspect（切面）** | 切入点 + 通知 | 日志切面、事务切面 |
| **Target（目标对象）** | 被代理的对象 | UserController |
| **Proxy（代理对象）** | AOP 生成的代理对象 | UserController 的代理 |
| **Weaving（织入）** | 将切面应用到目标对象的过程 | 编译期、类加载期、运行期 |

---

### AOP 通知类型

| 注解 | 说明 | 执行时机 |
|------|------|----------|
| `@Before` | 前置通知 | 目标方法执行前 |
| `@AfterReturning` | 返回后通知 | 目标方法成功返回后 |
| `@AfterThrowing` | 异常通知 | 目标方法抛出异常后 |
| `@After` | 后置通知 | 目标方法执行后（无论成功还是失败） |
| `@Around` | 环绕通知 | 目标方法执行前后（功能最强大） |

**执行顺序**：

```
@Around（前）
    ↓
@Before
    ↓
    目标方法执行
    ↓
@AfterReturning / @AfterThrowing
    ↓
@After
    ↓
@Around（后）
```

### 步骤4：创建 IP 工具类

创建 `com.example.demo.utils.IpUtils.java`：

```java
package com.example.demo.utils;

import jakarta.servlet.http.HttpServletRequest;

/**
 * IP 工具类
 */
public class IpUtils {

    private static final String UNKNOWN = "unknown";
    private static final String LOCALHOST_IP = "127.0.0.1";
    private static final String LOCALHOST_IP_16 = "0:0:0:0:0:0:0:1";
    private static final int IP_LEN = 15;

    /**
     * 获取客户端 IP 地址
     */
    public static String getIpAddr(HttpServletRequest request) {
        if (request == null) {
            return "unknown";
        }

        String ip = request.getHeader("x-forwarded-for");
        if (ip == null || ip.length() == 0 || UNKNOWN.equalsIgnoreCase(ip)) {
            ip = request.getHeader("Proxy-Client-IP");
        }
        if (ip == null || ip.length() == 0 || UNKNOWN.equalsIgnoreCase(ip)) {
            ip = request.getHeader("X-Forwarded-For");
        }
        if (ip == null || ip.length() == 0 || UNKNOWN.equalsIgnoreCase(ip)) {
            ip = request.getHeader("WL-Proxy-Client-IP");
        }
        if (ip == null || ip.length() == 0 || UNKNOWN.equalsIgnoreCase(ip)) {
            ip = request.getHeader("X-Real-IP");
        }
        if (ip == null || ip.length() == 0 || UNKNOWN.equalsIgnoreCase(ip)) {
            ip = request.getRemoteAddr();
        }

        // 对于通过多个代理的情况，第一个 IP 为客户端真实 IP，多个 IP 按照','分割
        if (ip != null && ip.length() > IP_LEN) {
            int index = ip.indexOf(",");
            if (index > 0) {
                ip = ip.substring(0, index);
            }
        }

        // 将 IPv6 的 localhost 转换为 IPv4
        if (LOCALHOST_IP_16.equals(ip)) {
            ip = LOCALHOST_IP;
        }

        return ip;
    }
}
```

**说明**：
- 获取真实客户端 IP（考虑代理情况）
- 优先级：`x-forwarded-for` > `Proxy-Client-IP` > `X-Forwarded-For` > `WL-Proxy-Client-IP` > `X-Real-IP` > `RemoteAddr`

### 步骤5：创建日志切面类

创建 `com.example.demo.aspect.LogAspect.java`：

```java
package com.example.demo.aspect;

import com.example.demo.annotation.Log;
import com.example.demo.dto.LogDTO;
import com.example.demo.utils.IpUtils;
import com.fasterxml.jackson.databind.ObjectMapper;
import lombok.extern.slf4j.Slf4j;
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.Around;
import org.aspectj.lang.annotation.Aspect;
import org.aspectj.lang.annotation.Pointcut;
import org.aspectj.lang.reflect.MethodSignature;
import org.springframework.stereotype.Component;
import org.springframework.web.context.request.RequestContextHolder;
import org.springframework.web.context.request.ServletRequestAttributes;

import jakarta.servlet.http.HttpServletRequest;
import java.lang.reflect.Method;
import java.util.Arrays;

/**
 * 操作日志切面
 */
@Slf4j
@Aspect
@Component
public class LogAspect {

    private final ObjectMapper objectMapper = new ObjectMapper();

    /**
     * 定义切入点：拦截所有带 @Log 注解的方法
     */
    @Pointcut("@annotation(com.example.demo.annotation.Log)")
    public void logPointcut() {
    }

    /**
     * 环绕通知
     */
    @Around("logPointcut()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        long startTime = System.currentTimeMillis();

        // 获取请求属性
        ServletRequestAttributes attributes = (ServletRequestAttributes) RequestContextHolder.getRequestAttributes();
        HttpServletRequest request = attributes != null ? attributes.getRequest() : null;

        // 创建日志对象
        LogDTO logDTO = new LogDTO();

        // 获取方法签名
        MethodSignature signature = (MethodSignature) joinPoint.getSignature();
        Method method = signature.getMethod();

        // 获取 @Log 注解
        Log logAnnotation = method.getAnnotation(Log.class);
        if (logAnnotation != null) {
            logDTO.setDescription(logAnnotation.value());
        }

        // 设置基本信息
        if (request != null) {
            logDTO.setIp(IpUtils.getIpAddr(request));
            logDTO.setUrl(request.getRequestURI());
            logDTO.setHttpMethod(request.getMethod());
        }

        logDTO.setMethod(signature.getDeclaringTypeName() + "." + signature.getName());
        logDTO.setParams(Arrays.toString(joinPoint.getArgs()));

        try {
            // 执行目标方法
            Object result = joinPoint.proceed();

            // 记录成功日志
            long endTime = System.currentTimeMillis();
            logDTO.setTime(endTime - startTime);
            logDTO.setSuccess(true);
            logDTO.setResult(objectMapper.writeValueAsString(result));

            log.info("操作日志: {}", logDTO);

            return result;
        } catch (Exception e) {
            // 记录失败日志
            long endTime = System.currentTimeMillis();
            logDTO.setTime(endTime - startTime);
            logDTO.setSuccess(false);
            logDTO.setErrorMsg(e.getMessage());

            log.error("操作异常: {}", logDTO, e);

            throw e;
        }
    }
}
```

**代码说明**：
- `@Pointcut` - 定义切入点，拦截带 `@Log` 注解的方法
- `@Around` - 环绕通知，可以在方法执行前后添加逻辑
- `ProceedingJoinPoint` - 连接点，可以获取目标方法的信息
- `RequestContextHolder` - 获取当前请求的上下文

## 第三部分：Pointcut 表达式详解（20分钟）

### Pointcut 表达式语法

```
execution(访问修饰符 返回类型 包名.类名.方法名(参数类型) throws 异常类型)
```

**通配符**：
- `*` - 匹配任意字符（通常用于一个部分）
- `..` - 匹配任意字符（通常用于多个部分，如包或参数）
- `+` - 匹配子类

**语法示例**：
```
execution(* com.example.demo.controller.UserController.getName(..))
            ↓           ↓                       ↓           ↓    ↓
          返回类型      包名                    类名        方法名 参数
```

---

### Pointcut 的本质：什么是切入点？

#### 什么是 Pointcut？

**Pointcut（切入点）**：定义"在哪些地方"执行切面逻辑。

**打个比方**：
- **Pointcut** = 摄像头的位置（在哪里安装监控）
- **Advice（通知）** = 摄像头录制的内容（监控要做什么）
- **Aspect（切面）** = 完整的监控系统（摄像头 + 录制功能）

```
┌─────────────────────────────────────────────────┐
│  Aspect（切面）                                  │
│  ┌─────────────────┐      ┌─────────────────┐  │
│  │ Pointcut        │ +    │ Advice          │  │
│  │ （在哪里执行）   │      │ （执行什么）     │  │
│  └─────────────────┘      └─────────────────┘  │
└─────────────────────────────────────────────────┘
```

#### 为什么要用 Pointcut？

**场景1：不用 Pointcut 的问题**

假设要为所有 Controller 方法记录日志：

```java
// ❌ 没有使用 Pointcut：每个方法都要写一遍
@RestController
public class UserController {

    @GetMapping("/user/{id}")
    public User getUser(@PathVariable Long id) {
        log.info("开始执行：getUser");  // 重复代码
        log.info("参数：id={}", id);    // 重复代码
        User user = userService.getById(id);
        log.info("执行结束：getUser");  // 重复代码
        return user;
    }

    @PostMapping("/user")
    public User createUser(@RequestBody User user) {
        log.info("开始执行：createUser");  // 重复代码
        log.info("参数：user={}", user);   // 重复代码
        User result = userService.create(user);
        log.info("执行结束：createUser");  // 重复代码
        return result;
    }

    @DeleteMapping("/user/{id}")
    public void deleteUser(@PathVariable Long id) {
        log.info("开始执行：deleteUser");  // 重复代码
        log.info("参数：id={}", id);       // 重复代码
        userService.delete(id);
        log.info("执行结束：deleteUser");  // 重复代码
    }

    // ... 100 个方法，就要写 300 次日志代码！
}
```

**问题**：
- 代码重复：每个方法都要写相同的日志代码
- 维护困难：如果要修改日志格式，要改 100 个地方
- 容易遗漏：某个方法忘记加日志怎么办？
- 业务混乱：业务代码和日志代码混在一起

---

**场景2：使用 Pointcut 的优势**

```java
// ✅ 使用 Pointcut：一次定义，到处生效

// 1. 定义切面（只需要写一次）
@Aspect
@Component
@Slf4j
public class LoggingAspect {

    // 定义 Pointcut：匹配所有 Controller 方法
    @Pointcut("execution(* com.example.demo.controller.*.*(..))")
    public void controllerMethods() {}

    // 定义 Advice：在方法执行前记录日志
    @Before("controllerMethods()")
    public void logBefore(JoinPoint joinPoint) {
        log.info("开始执行：{}", joinPoint.getSignature().getName());
        log.info("参数：{}", Arrays.toString(joinPoint.getArgs()));
    }

    // 定义 Advice：在方法执行后记录日志
    @After("controllerMethods()")
    public void logAfter(JoinPoint joinPoint) {
        log.info("执行结束：{}", joinPoint.getSignature().getName());
    }
}

// 2. Controller 代码（不需要写任何日志代码）
@RestController
public class UserController {

    @GetMapping("/user/{id}")
    public User getUser(@PathVariable Long id) {
        return userService.getById(id);  // 自动会有日志
    }

    @PostMapping("/user")
    public User createUser(@RequestBody User user) {
        return userService.create(user);  // 自动会有日志
    }

    @DeleteMapping("/user/{id}")
```

```java
@Aspect
```

```java
public class PerformanceAspect {
```

#### 场景3：事务管理

**需求**：所有修改数据库的方法都要开启事务

```java
@Aspect
@Component
public class TransactionAspect {

    // Pointcut：匹配所有修改操作（insert、update、delete）
    @Pointcut("execution(* com.example.demo.service.*.insert*(..)) || " +
              "execution(* com.example.demo.service.*.update*(..)) || " +
              "execution(* com.example.demo.service.*.delete*(..))")
    public void modifyMethods() {}

    // Advice：环绕通知，管理事务
    @Around("modifyMethods()")
    public Object manageTransaction(ProceedingJoinPoint joinPoint) throws Throwable {
        TransactionStatus status = transactionManager.getTransaction(new DefaultTransactionDefinition());

        try {
            Object result = joinPoint.proceed();  // 执行业务方法
            transactionManager.commit(status);    // 提交事务
            return result;
        } catch (Exception e) {
            transactionManager.rollback(status);  // 回滚事务
            throw e;
        }
    }
}
```

#### 场景4：缓存管理

**需求**：对查询方法的结果进行缓存

```java
@Aspect
@Component
public class CacheAspect {

    private final Map<String, Object> cache = new ConcurrentHashMap<>();

    // Pointcut：匹配所有查询方法（find、get、query、select）
    @Pointcut("execution(* com.example.demo.service.*.find*(..)) || " +
              "execution(* com.example.demo.service.*.get*(..)) || " +
              "execution(* com.example.demo.service.*.query*(..))")
    public void queryMethods() {}

    // Advice：环绕通知，实现缓存
    @Around("queryMethods()")
    public Object cacheResult(ProceedingJoinPoint joinPoint) throws Throwable {
        // 生成缓存 key
        String key = joinPoint.getSignature().toString() +
                     Arrays.toString(joinPoint.getArgs());

        // 检查缓存
        Object cachedResult = cache.get(key);
        if (cachedResult != null) {
            log.info("从缓存返回：{}", key);
            return cachedResult;
        }

        // 执行原方法
        Object result = joinPoint.proceed();

        // 存入缓存
        cache.put(key, result);
        log.info("存入缓存：{}", key);

        return result;
    }
}
```

### 什么时候使用 Pointcut？

#### ✅ 应该使用 Pointcut 的场景

| 场景 | 示例 | 说明 |
|------|------|------|
| **横切关注点** | 日志、权限、事务、缓存 | 这些功能跨越多个模块，需要在多处使用 |
| **重复代码** | 每个方法都记录日志 | 如果同样代码出现 3 次以上，考虑用 AOP |
| **统一规则** | 所有 Controller 方法都要记录日志 | 需要对一批方法应用相同规则 |
| **非业务需求** | 性能监控、异常处理 | 与业务逻辑无关，属于系统层面的需求 |

#### ❌ 不应该使用 Pointcut 的场景

| 场景 | 原因 |
|------|------|
| **业务特定逻辑** | 只在一个方法中使用的逻辑 |
| **简单方法** | 只有两三个方法需要处理 |
| **强依赖上下文** | 需要访问方法内部变量 |
| **性能敏感** | AOP 有少量性能开销 |

### Pointcut 设计原则

#### 1. 精确原则

```java
// ❌ 过于宽泛：可能匹配到不想要的方法
@Pointcut("execution(* get*(..))")

// ✅ 精确匹配：明确指定包和类
@Pointcut("execution(* com.example.demo.controller.PointcutTestController.get*(..))")
```

#### 2. 可读性原则

```java
// ❌ 难以理解
@Pointcut("execution(*(..
com.*.*(..))&&@annotation(*)")

// ✅ 清晰明了
@Pointcut("execution(* com.example.demo.controller.*.*(..))")
@Pointcut("@annotation(org.springframework.web.bind.annotation.GetMapping)")
@Pointcut("controllerMethods() && getMappings()")
```

#### 3. 可维护性原则

```java
// ❌ 硬编码表达式，难以复用
@Before("execution(* com.example.demo.controller.*.*(..))")

// ✅ 使用命名 Pointcut，易于复用
@Pointcut("execution(* com.example.demo.controller.*.*(..))")
public void controllerMethods() {}

@Before("controllerMethods()")
public void logBefore() {}
```

### 总结：Pointcut 的价值

| 维度 | 不用 Pointcut | 使用 Pointcut |
|------|--------------|--------------|
| **代码量** | 100 个方法 × 10 行 = 1000 行 | 1 个切面 × 20 行 = 20 行 |
| **维护成本** | 修改日志格式要改 100 个地方 | 只需改 1 个地方 |
| **遗漏风险** | 容易忘记加日志 | 自动应用到所有匹配方法 |
| **代码质量** | 业务代码和横切逻辑混在一起 | 关注点分离，职责清晰 |
| **测试** | 每个方法都要测试日志逻辑 | 日志逻辑独立测试 |

**核心思想**：
- **Pointcut** 让你把"在哪些地方执行"和"执行什么"分离
- 一个 Pointcut 可以被多个 Advice 复用
- 组合 Pointcut 可以实现复杂的匹配规则

### Pointcut 表达式总结

| 练习 | 表达式 | 说明 | 何时匹配 |
|------|--------|------|----------|
| 练习1 | `execution(* com.example.demo.controller.PointcutTestController.*(..))` | 特定类 | PointcutTestController 的所有方法 |
| 练习2 | `execution(* com.example.demo.controller.*.*(..))` | 特定包 | controller 包下所有类的所有方法 |
| 练习3 | `execution(* com.example.demo.controller.PointcutTestController.*())` | 无参数 | PointcutTestController 中不带参数的方法 |
| 练习4 | `execution(* com.example.demo.controller.PointcutTestController.*(String))` | 单参数 | PointcutTestController 中带一个 String 参数的方法 |
| 练习5 | `execution(* com.example.demo.controller.PointcutTestController.*(String, Integer))` | 双参数 | PointcutTestController 中带两个参数的方法 |
| 练习6 | `execution(java.lang.String com.example.demo.controller.PointcutTestController.*(..))` | 返回类型 | PointcutTestController 中返回 String 的方法 |
| 练习7 | `execution(* com.example.demo.controller.PointcutTestController.get*(..))` | 方法名前缀 | PointcutTestController 中以 get 开头的方法 |
| 练习8 | `@annotation(GetMapping)` | 注解 | 带 @GetMapping 注解的方法 |
| 练习9 | `pointcut1() && pointcut2()` | 逻辑与 | 同时满足两个条件 |
| 练习10 | `@annotation(PostMapping) \|\| @annotation(PutMapping)` | 逻辑或 | 满足任一条件 |

**重要提示**：
- ⚠️ **避免使用过于宽泛的表达式**（如 `execution(* get*(..))`）
- 过于宽泛的表达式会匹配到 Spring 内部组件，导致系统异常
- **始终指定完整的包名和类名**，限制 Pointcut 的作用范围

---

### 常见错误示例

#### 错误1：使用过于宽泛的表达式（⚠️ 最严重）

```java
// ❌ 危险：匹配项目中所有以 get 开头的方法，包括 Spring 内部组件！
@Pointcut("execution(* get*(..))")

// ✅ 正确：限制在特定的类中
@Pointcut("execution(* com.example.demo.controller.PointcutTestController.get*(..))")
```

**危害**：
- 过于宽泛的表达式会匹配到 Spring 内部 Bean 的方法
- 会破坏 Spring 容器的初始化过程
- 导致 `NullPointerException` 和各种异常
- **系统无法启动**

---

#### 错误2：包路径中使用了单点

```java
// ❌ 错误：只匹配 com.example.demo.controller 下的类，不包含子包
@Pointcut("execution(* com.example.demo.controller.*.*(..))")

// ✅ 正确：匹配 com.example.demo.controller 及其所有子包
@Pointcut("execution(* com.example.demo.controller..*.*(..))")
```

**区别**：
- 单点 `.` 只匹配当前层级
- 双点 `..` 匹配当前层级及所有子层级

#### 错误3：参数类型写法错误

```java
// ❌ 错误：参数类型应该是全限定名或简单类型，不能带包名缩写
@Pointcut("execution(* *(String))")  // 如果有多个 String 类会冲突

// ✅ 正确：使用完整类名（当有歧义时）
@Pointcut("execution(* *(java.lang.String))")
```

#### 错误4：返回类型写法错误

```java
// ❌ 错误：返回类型必须写完整
@Pointcut("execution(String *(..))")

// ✅ 正确：使用完整类名
@Pointcut("execution(java.lang.String *(..))")
```

### 创建一个新的切面用于演示

创建 `com.example.demo.aspect.DemoAspect.java`：

```java
package com.example.demo.aspect;

import lombok.extern.slf4j.Slf4j;
import org.aspectj.lang.JoinPoint;
import org.aspectj.lang.ProceedingJoinPoint;
import org.aspectj.lang.annotation.*;
import org.springframework.stereotype.Component;

/**
 * 演示切面（各种通知类型）
 */
@Slf4j
@Aspect
@Component
public class DemoAspect {

    /**
     * 定义切入点
     */
    @Pointcut("execution(* com.example.demo.controller.UserController.*(..))")
    public void controllerPointcut() {}

    /**
     * 前置通知：在目标方法执行前执行
     */
    @Before("controllerPointcut()")
    public void before(JoinPoint joinPoint) {
        log.info("【前置通知】方法准备执行: {}", joinPoint.getSignature());
    }

    /**
     * 后置通知：在目标方法执行后执行（无论成功还是失败）
     */
    @After("controllerPointcut()")
    public void after(JoinPoint joinPoint) {
        log.info("【后置通知】方法执行完毕: {}", joinPoint.getSignature());
    }

    /**
     * 返回通知：在目标方法成功返回后执行
     */
    @AfterReturning(pointcut = "controllerPointcut()", returning = "result")
    public void afterReturning(JoinPoint joinPoint, Object result) {
        log.info("【返回通知】方法成功返回: {}, 返回值: {}", joinPoint.getSignature(), result);
    }

    /**
     * 异常通知：在目标方法抛出异常后执行
     */
    @AfterThrowing(pointcut = "controllerPointcut()", throwing = "exception")
    public void afterThrowing(JoinPoint joinPoint, Exception exception) {
        log.error("【异常通知】方法抛出异常: {}, 异常: {}", joinPoint.getSignature(), exception.getMessage());
    }

    /**
     * 环绕通知：在目标方法执行前后都执行（最强大的通知类型）
     *
     * 特点：
     * 1. 可以控制目标方法是否执行
     * 2. 可以修改目标方法的参数
     * 3. 可以修改目标方法的返回值
     * 4. 可以捕获/处理异常
     * 5. 可以统计方法执行时间
     */
    @Around("controllerPointcut()")
    public Object around(ProceedingJoinPoint joinPoint) throws Throwable {
        String methodName = joinPoint.getSignature().getName();

        // 前置逻辑
        log.info("【环绕通知-前置】方法准备执行: {}", methodName);
        long startTime = System.currentTimeMillis();

        Object result = null;
        try {
            // 执行目标方法（必须调用，否则目标方法不会执行）
            result = joinPoint.proceed();

            // 后置逻辑（方法成功返回）
            long endTime = System.currentTimeMillis();
            long executionTime = endTime - startTime;
            log.info("【环绕通知-后置】方法执行成功: {}, 耗时: {}ms", methodName, executionTime);

            return result;
        } catch (Exception e) {
            // 异常逻辑（方法抛出异常）
            log.error("【环绕通知-异常】方法执行失败: {}, 异常: {}", methodName, e.getMessage());
            throw e;  // 可以选择抛出或处理异常
        } finally {
            // 最终逻辑（无论成功还是失败都会执行）
            log.info("【环绕通知-最终】方法执行结束: {}", methodName);
        }
    }
}
```

### 各种通知类型对比

| 通知类型 | 注解 | 执行时机 | 能否获取返回值 | 能否修改返回值 | 能否阻止方法执行 | 应用场景 |
|---------|------|----------|--------------|--------------|----------------|---------|
| **前置通知** | @Before | 目标方法执行前 | ❌ | ❌ | ❌ | 权限检查、参数校验、日志记录 |
| **后置通知** | @After | 目标方法执行后（无论成功失败） | ❌ | ❌ | ❌ | 释放资源、记录完成状态 |
| **返回通知** | @AfterReturning | 目标方法成功返回后 | ✅ | ❌ | ❌ | 记录返回值、缓存结果 |
| **异常通知** | @AfterThrowing | 目标方法抛出异常后 | ❌ | ❌ | ❌ | 异常监控、发送告警 |
| **环绕通知** | @Around | 目标方法执行前后 | ✅ | ✅ | ✅ | 性能监控、事务管理、缓存 |

**关键区别**：
- @Around 是最强大的通知类型
- @Around 可以完全控制方法的执行
- 其他通知类型只能"观察"，不能"干预"

### 环绕通知的高级用法

#### 用法1：性能监控

```java
public Object logExecutionTime(ProceedingJoinPoint joinPoint) throws Throwable {

    Object result = joinPoint.proceed();

    long end = System.currentTimeMillis();
    log.info("{} 执行时间: {}ms", joinPoint.getSignature(), end - start);

    return result;
}
```

```java
@Around("queryMethods()")
public Object cacheResult(ProceedingJoinPoint joinPoint) throws Throwable {
    String key = generateKey(joinPoint);

    // 检查缓存
    Object cachedResult = cache.get(key);
    if (cachedResult != null) {
        log.info("从缓存返回: {}", key);
        return cachedResult;
    }

    // 执行原方法
    Object result = joinPoint.proceed();

    // 存入缓存
    cache.put(key, result);
    return result;
}
```

### 执行顺序总结

**正常执行时的完整顺序**：
```
1. @Around（前置部分）
2. @Before
3. 目标方法执行
4. @Around（后置部分，try 块）
5. @AfterReturning
6. @After
7. @Around（finally 块）
```

**异常执行时的完整顺序**：
```
1. @Around（前置部分）
2. @Before
3. 目标方法执行（抛出异常）
4. @Around（异常部分，catch 块）
5. @Around（finally 块）
6. @AfterThrowing
7. @After
```

**重要提醒**：
- @Around 的前置逻辑最先执行
- @Around 的 finally 块在 @AfterThrowing 和 @After 之前执行
- 同一切面内，通知顺序由注解类型决定
- 不同切面间，可用 @Order 指定顺序

## 第五部分：跨域问题与解决方案（30分钟）

### 什么是跨域？

**跨域（Cross-Origin）**：浏览器安全策略，限制一个域下的文档或脚本与另一个域的资源进行交互。

---

### 同源策略

**同源**：协议、域名、端口完全相同。

| URL1 | URL2 | 是否同源 | 原因 |
|------|------|----------|------|
| `http://example.com/api` | `http://example.com/api` | ✅ 是 | 完全相同 |
| `http://example.com/api` | `http://example.com/user` | ✅ 是 | 只有路径不同 |
| `http://example.com/api` | `https://example.com/api` | ❌ 否 | 协议不同 |
| `http://example.com/api` | `http://www.example.com/api` | ❌ 否 | 域名不同 |
| `http://example.com:8080/api` | `http://example.com:8081/api` | ❌ 否 | 端口不同 |

---

### 跨域场景示例

**前端**：`http://localhost:5173`（Vue 开发服务器）
**后端**：`http://localhost:8080`（Spring Boot）

访问时浏览器会报错：

```
Access to XMLHttpRequest at 'http://localhost:8080/user/list'
from origin 'http://localhost:5173' has been blocked by CORS policy:
No 'Access-Control-Allow-Origin' header is present on the requested resource.
```

---

### CORS（跨域资源共享）工作原理

#### 简单请求

浏览器直接发送请求，服务器在响应头中添加：
```
Access-Control-Allow-Origin: http://localhost:5173
```

---

#### 复杂请求（预检）

浏览器先发送 `OPTIONS` 请求（预检），服务器返回允许的头部和方法：

```http
OPTIONS /user/register HTTP/1.1
Host: localhost:8080
Origin: http://localhost:5173
Access-Control-Request-Method: POST
Access-Control-Request-Headers: Content-Type
```

服务器响应：

```http
HTTP/1.1 200 OK
Access-Control-Allow-Origin: http://localhost:5173
Access-Control-Allow-Methods: POST, GET, OPTIONS
Access-Control-Allow-Headers: Content-Type, Authorization
Access-Control-Max-Age: 3600
```

---

### 解决方案

#### 方案1：使用 @CrossOrigin 注解（简单）

在 Controller 或方法上添加 `@CrossOrigin` 注解。

```java
@RestController
@RequestMapping("/user")
@CrossOrigin(origins = "http://localhost:5173")  // 类级别
public class UserController {

    @GetMapping("/list")
    @CrossOrigin(origins = "*")  // 方法级别（覆盖类级别）
    public Result<List<User>> getUserList() {
        // ...
    }
}
```

**配置项**：
- `origins` - 允许的源（如 `http://localhost:5173` 或 `*` 表示允许所有）
- `methods` - 允许的 HTTP 方法
- `allowedHeaders` - 允许的请求头
- `exposedHeaders` - 暴露的响应头
- `allowCredentials` - 是否允许携带凭证
- `maxAge` - 预检请求缓存时间（秒）

---

#### 方案2：全局 CORS 配置（推荐）

创建 `com.example.demo.config.CorsConfig.java`：

```java
package com.example.demo.config;

import org.springframework.context.annotation.Bean;
import org.springframework.context.annotation.Configuration;
import org.springframework.web.cors.CorsConfiguration;
import org.springframework.web.cors.UrlBasedCorsConfigurationSource;
import org.springframework.web.filter.CorsFilter;

/**
 * CORS 跨域配置
 */
@Configuration
public class CorsConfig {

    @Bean
    public CorsFilter corsFilter() {
        CorsConfiguration config = new CorsConfiguration();

        // 允许所有来源（生产环境应指定具体域名）
        config.addAllowedOriginPattern("*");

        // 允许携带凭证
        config.setAllowCredentials(true);

        // 允许所有请求方法
        config.addAllowedMethod("*");

        // 允许所有请求头
        config.addAllowedHeader("*");

        // 暴露的响应头
        config.addExposedHeader("Content-Disposition");

        // 预检请求缓存时间（秒）
        config.setMaxAge(3600L);

        // 映射路径
        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);

        return new CorsFilter(source);
    }
}
```

---

#### 方案3：使用 WebMvcConfigurer（推荐）

创建 `com.example.demo.config.WebMvcConfig.java`：

```java
package com.example.demo.config;

import org.springframework.context.annotation.Configuration;
import org.springframework.web.servlet.config.annotation.CorsRegistry;
import org.springframework.web.servlet.config.annotation.WebMvcConfigurer;

/**
 * Web MVC 配置
 */
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {

    /**
     * 配置跨域
     */
    @Override
    public void addCorsMappings(CorsRegistry registry) {
        registry.addMapping("/**")  // 对所有路径生效
                .allowedOriginPatterns("*")  // 允许所有来源
                .allowedMethods("GET", "POST", "PUT", "DELETE", "OPTIONS")  // 允许的方法
                .allowedHeaders("*")  // 允许的请求头
                .exposedHeaders("Content-Disposition")  // 暴露的响应头
                .allowCredentials(true)  // 允许携带凭证
                .maxAge(3600);  // 预检缓存时间
    }
}
```

**说明**：
- 实现了 `WebMvcConfigurer` 接口
- `addCorsMappings` 方法配置 CORS 规则
- 这种方式更灵活，可以针对不同路径配置不同规则

---

#### 方案4：配置文件方式（Spring Boot 2.4+）

在 `application.yml` 中配置：

```yaml
spring:
  web:
    cors:
      mappings:
        /**:
          allowed-origins: "http://localhost:5173,http://localhost:3000"
          allowed-methods: "GET,POST,PUT,DELETE,OPTIONS"
          allowed-headers: "*"
          exposed-headers: "Content-Disposition"
          allow-credentials: true
          max-age: 3600
```

### 配置对比

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| `@CrossOrigin` | 简单直接 | 需要每个 Controller 添加 | 小型项目 |
| `CorsFilter` | 功能强大 | 配置较复杂 | 需要细粒度控制 |
| `WebMvcConfigurer` | 灵活、推荐 | 需要实现接口 | **推荐使用** |
| 配置文件 | 集中管理 | Spring Boot 2.4+ 支持 | 简单场景 |

### 日志 vs System.out.println

| 对比项 | System.out.println | 日志框架（Logback） |
|--------|-------------------|-------------------|
| **性能** | 每次都打印（即使不需要） | 按需打印，性能更好 |

| **输出目标** | 仅控制台 | 控制台、文件、数据库、远程服务器 |

### Logback 简介

**Logback** 是 Java 的日志框架，Spring Boot 默认集成了 Logback。

**日志级别**（从低到高）：

```
TRACE < DEBUG < INFO < WARN < ERROR
```

| 级别 | 用途 | 示例 |
|------|------|------|
| **TRACE** | 最详细的信息（一般不用） | 方法入参、出参的详细值 |
| **DEBUG** | 调试信息 | SQL 语句、中间变量值 |
| **INFO** | 重要信息 | 用户登录、订单创建 |
| **WARN** | 警告信息（不影响运行） | 配置项使用默认值、缓存未命中 |
| **ERROR** | 错误信息 | 异常、失败的操作 |

**级别规则**：
- 设置为 `INFO`，则只打印 INFO、WARN、ERROR
- 设置为 `DEBUG`，则打印 DEBUG、INFO、WARN、ERROR
- **开发环境**：使用 `DEBUG`（看到更多信息）
- **生产环境**：使用 `INFO` 或 `WARN`（减少日志量）

---

### Logback 的核心组件

#### 1. Logger（日志记录器）

负责记录日志信息，可以设置不同的日志级别。

```java
// 在代码中使用
@Slf4j  // Lombok 注解，自动生成 log 对象
public class UserService {
    public void createUser(User user) {
        log.debug("创建用户，参数：{}", user);  // DEBUG 级别
        log.info("用户创建成功，用户名：{}", user.getUsername());  // INFO 级别
        log.warn("邮箱未验证，用户名：{}", user.getUsername());  // WARN 级别
        log.error("用户创建失败", e);  // ERROR 级别
    }
}
```

#### 2. Appender（输出目的地）

控制日志输出到哪里。

| Appender | 作用 | 使用场景 |
|---------|------|----------|
| **ConsoleAppender** | 输出到控制台 | 开发环境、本地调试 |
| **FileAppender** | 输出到文件 | 记录日志到文件 |
| **RollingFileAppender** | 滚动文件输出 | 生产环境（自动切分） |
| **SocketAppender** | 输出到远程服务器 | 分布式日志收集 |

#### 3. Layout/Encoder（格式化）

控制日志的输出格式。

```log
2026-02-01 10:23:45.123 [http-nio-8080-exec-5] INFO  c.e.d.service.UserService - 用户创建成功，用户名：zhangsan
↑时间             ↑线程                                  ↑级别 ↑类名              ↑消息
```

### 日志最佳实践

#### 1. 选择合适的日志级别

```java
// ❌ 错误：所有日志都用 INFO
log.info("开始查询数据库");
log.info("SQL: SELECT * FROM user WHERE id = 1");
log.info("查询结果：User(id=1, name=zhangsan)");

// ✅ 正确：使用合适的级别
log.debug("开始查询数据库");  // DEBUG：调试信息
log.debug("SQL: SELECT * FROM user WHERE id = {}", id);  // DEBUG：SQL 语句
log.info("查询用户成功，用户名：{}", user.getName());  // INFO：业务关键信息
```

#### 2. 使用占位符（不要字符串拼接）

```java
// ❌ 错误：字符串拼接（即使不打印也会拼接，性能差）
log.info("用户登录，用户名：" + username + "，IP：" + ip);

// ✅ 正确：使用占位符（只在需要打印时才拼接）
log.info("用户登录，用户名：{}，IP：{}", username, ip);
```

#### 3. 记录关键业务信息

```java
// ✅ 记录用户操作
log.info("用户[{}]创建了订单[{}]，金额：{}", userId, orderId, amount);

// ✅ 记录异常堆栈
log.error("支付失败，订单号：{}", orderId, e);  // 注意：最后传异常对象

// ❌ 不要记录敏感信息
log.info("用户登录，密码：{}", password);  // 危险！不要记录密码
log.info("支付，信用卡号：{}", creditCard);  // 危险！不要记录完整卡号
```

#### 4. 生产环境配置建议

```xml
<!-- 生产环境日志级别 -->
<root level="INFO">
    <appender-ref ref="CONSOLE"/>
    <appender-ref ref="FILE"/>
</root>

<!-- 应用日志（生产环境用 INFO） -->
<logger name="com.example.demo" level="INFO"/>

<!-- 框架日志（减少噪音） -->
<logger name="org.springframework" level="WARN"/>
<logger name="org.hibernate" level="WARN"/>
```

### 配置 logback

在 `src/main/resources` 下创建 `logback-spring.xml`：

```xml
<?xml version="1.0" encoding="UTF-8"?>
<configuration>
    <!-- 定义日志输出格式 -->
    <property name="CONSOLE_LOG_PATTERN"
              value="%d{yyyy-MM-dd HH:mm:ss.SSS} [%thread] %-5level %logger{36} - %msg%n"/>

    <!-- 控制台输出 -->
    <appender name="CONSOLE" class="ch.qos.logback.core.ConsoleAppender">
        <encoder>
            <pattern>${CONSOLE_LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>
    </appender>

    <!-- 文件输出 -->
    <appender name="FILE" class="ch.qos.logback.core.rolling.RollingFileAppender">
        <file>logs/application.log</file>
        <encoder>
            <pattern>${CONSOLE_LOG_PATTERN}</pattern>
            <charset>UTF-8</charset>
        </encoder>

        <!-- 滚动策略 -->
        <rollingPolicy class="ch.qos.logback.core.rolling.TimeBasedRollingPolicy">
            <fileNamePattern>logs/application-%d{yyyy-MM-dd}.%i.log</fileNamePattern>
            <timeBasedFileNamingAndTriggeringPolicy class="ch.qos.logback.core.rolling.SizeAndTimeBasedFNATP">
                <maxFileSize>10MB</maxFileSize>
            </timeBasedFileNamingAndTriggeringPolicy>
            <maxHistory>30</maxHistory>
        </rollingPolicy>
    </appender>

    <!-- 日志级别 -->
    <root level="INFO">
        <appender-ref ref="CONSOLE"/>
        <appender-ref ref="FILE"/>
    </root>

    <!-- 应用日志 -->
    <logger name="com.example.demo" level="DEBUG"/>

    <!-- 框架日志 -->
    <logger name="org.springframework" level="INFO"/>
    <logger name="org.hibernate" level="INFO"/>
</configuration>
```

## 常见问题解答

### Q1：AOP 不生效怎么办？

**可能原因**：
1. 忘记添加 `@Component` 或 `@Aspect` 注解
2. 切面类不在 Spring 扫描路径下
3. 切入点表达式写错了
4. 方法是 `private` 或 `final`（AOP 基于代理，无法拦截）
5. 同类内部方法调用（不经过代理）

**解决方案**：
```java
// 错误：同类内部调用
@Service
public class UserService {
    public void methodA() {
        methodB();  // 不会触发 AOP
    }

    @Log
    public void methodB() {
        // ...
    }
}

// 正确：注入自己调用
@Service
public class UserService {
    @Autowired
    private UserService self;  // 注入代理对象

    public void methodA() {
        self.methodB();  // 会触发 AOP
    }
}
```

---

### Q2：多个切面的执行顺序？

**默认**：不确定（取决于类加载顺序）

**指定顺序**：使用 `@Order` 注解，值越小优先级越高。

```java
@Aspect
@Component
@Order(1)  // 先执行
public class LogAspect {
    // ...
}

@Aspect
@Component
@Order(2)  // 后执行
public class TransactionAspect {
    // ...
}
```

**环绕通知的执行顺序**：

```
@Order(1) @Around（前）
    ↓
@Order(2) @Around（前）
    ↓
目标方法
    ↓
@Order(2) @Around（后）
    ↓
@Order(1) @Around（后）
```

---

### Q3：CORS 配置了但还是报跨域错误？

**检查清单**：
1. 确认 CORS 配置的 `allowedOrigins` 包含前端地址
2. 确认没有使用 `allowCredentials(true)` + `allowedOrigins("*")`（不兼容）
3. 检查前端是否设置了 `withCredentials: true`
4. 检查是否有安全过滤器（如 Spring Security）干扰
5. 查看浏览器控制台的具体错误信息

**正确配置**：
```java
// 允许携带凭证时，必须指定具体域名
config.setAllowCredentials(true);
config.addAllowedOrigin("http://localhost:5173");  // 不能用 "*"

// 或者不允许携带凭证，允许所有域名
config.setAllowCredentials(false);
config.addAllowedOriginPattern("*");
```

### Q4：如何记录 POST 请求的 Body？

**问题**：流只能读取一次，读取后就无法传递给 Controller。

**方案**：使用 `ContentCachingRequestWrapper` 包装请求。

```java
@Component
public class RequestFilter extends OncePerRequestFilter {

    @Override
    protected void doFilterInternal(HttpServletRequest request, HttpServletResponse response, FilterChain filterChain) {
        ContentCachingRequestWrapper wrappedRequest = new ContentCachingRequestWrapper(request);
        filterChain.doFilter(wrappedRequest, response);

        // 读取 Body
        String body = new String(wrappedRequest.getContentAsByteArray());
        log.info("Request Body: {}", body);
    }
}
```

### Q5：AOP 会影响性能吗？

**答案**：会有一定影响，但很小。

**优化建议**：
1. 使用 `@Pointcut` 复用切入点
2. 避免在切面中执行耗时操作（如网络请求）
3. 使用 `@Async` 异步处理日志
4. 合理使用日志级别（生产环境使用 INFO 或 WARN）
